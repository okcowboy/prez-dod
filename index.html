DOD

Agile tour 2015 / Sophia antipolis

Historique

But

  • Obtenir un consensus autour du travail « terminé »
  • Fixer une liste de règles auditable visant la qualité
  • Minimiser les délais du risque
  • Améliorer la qualité des éléments produits par l’équipe
  • Rendre visible la qualité aux acteurs du projet
Mais comment?

Qualité Fonctionnelle

A-t-on construit la bonne solution?

  • Bonne définition des acceptances criterias
  • Des AC testés par les développeurs
  • Test d’intégration fonctionnel automatisé
  • Validation de la user story/AC par le PO
  • Testé par le métier

Qualité Technique

A-t-on bien construit?

    • Le développement
    • terminé
    • Build est au vert sur la Plateforme d’intégration continue
    • Revue de code
    • Robustesse/vulnérabilité
    • Commentaires
    • Cohérence des Tests unitaires

  • Documentation

    • Diagrammes / API
    • MAJ architecture / framework
    • MAJ de l’user story
  • Ticketing / Time tracking
    • Passage à DONE
    • Mise à jour de la charge
    • Changement d'état / Workflow

Qualité Technique

    • Les tests
    • Tests unitaires réalisés
    • Tests unitaires existants sans erreurs
    • Tests de performance
    • Tests d'accessibilité
    • Tests d’environnements (ie)
    • Mesure de la couverture de code
    • Mesure de la complexité cyclomatique

Préparer son DOD

    • Quand ?
    • Avant le début du premier Sprint
    • Avant la création des stories
    • Qui ?
    • Toute l'équipe!
    • Pourquoi ?
    • Transparence
    • Uniformisation des bonnes pratiques
    • Bonnes idées!
    • On n’impose pas le DOD!
